# GAPSCOPE — Project Instructions

This file is the starting reference for each Claude conversation. It captures the current state of the project so future chats have context. **Update it at the end of every working session.**

## What this project is

GAPSCOPE is a **capture integrity inspector**: a single-file, browser-based tool that analyzes timestamped data captures for timing anomalies — dropped samples, dropout locations, and arrivals that are too close together. It reads CSV/text captures, Harp `.bin` binary captures, and whole **experiment folders** (one sub-folder per device). It runs entirely client-side — no uploads, no server, no build step.

The whole application lives in **one file**: [index.html](index.html) (HTML + CSS + vanilla JS, no dependencies). Two Node harnesses, [test_experiment.js](test_experiment.js) (unit-level) and [test_flow.js](test_flow.js) (end-to-end over the sample experiment), run the page's real script with DOM stubs — `node test_experiment.js && node test_flow.js`.

Sample data in the repo: `2026-06-12T010203Z/` (a real experiment folder: 13 device dirs, root config files) and `devices/` (vendored known-device schemas, `<device>.device.yml`).

## Core concepts

- **Stream-type classification** — every stream is classed `pontual | constant | burst` by `classifyStream(gaps, Δt, tol, n, spanNs)` (built on `streamRegularity`). **Pontual**: on-beat fraction (gaps within `Δt/tol … Δt·tol`) `< CLASS_P_MIN` (0.65) — no dominant cadence, so Δt/missing are suppressed entirely. Otherwise **coverage** `= n·Δt / spanNs` decides: `> CLASS_COVER_MIN` (0.5) ⇒ **constant** (today's model — every gap > tol·Δt is a dropout), else ⇒ **burst** (regular cadence over only part of the timeline). For burst, missing is counted **only within runs**: in-data inter-burst silences (a bimodal gap-distribution jump ≥ `CLASS_JUMP`=8× above a `CLASS_BREAK_MULT`=12×Δt floor, `silenceNs`) are excluded, and the empty timeline before/after the data isn't a gap. `streamType` (per-stream override: auto/pontual/constant/burst, reset on load) vs `streamClass` (resolved) vs `autoClass` (what auto decided). Shown as a color-coded chip (C/B/P) on cards and the stream selector.
- **Experiment timeline** — devices run concurrently, so a single stream is shown/classified against the **whole experiment span**, not its own. `computeTimeline` reads each stream's first+last timestamp (bin: header `firstTs` + `readHarpLastTs` tail read; CSV: `readCsvBounds` head/tail) → `experiment.timeline = {startNs, endNs, durationNs}` (epoch-shifted ns). On load, `refOriginNs` anchors `offsets` to `startNs` (so `offsets[0]` ≥ 0 — the lead-in), and `timelineNs` is the full view/classification span; flat (non-experiment) loads keep own-span behavior (`refOriginNs=null`). `analyze` separates `dataSpanNs` (active span, for rate/constant-expected) from `durationNs`=`timelineNs` (the view). Empty timeline (before/after data, or inside a silence) renders as a dark "no data" heatmap band + a trace note.
- **Δt (nominal interval)** — the expected spacing between samples. Auto-detected per load by `detectInterval`: the **median gap refined to the mean of the gaps within ±50% of it**. The trim makes it tick-aware — Harp timestamps tick every 32 µs, so a true 1 ms register yields only 992/1024 µs gaps (~3:1) and the bare median locks onto 992 while the trimmed mean lands on the true 1000 (fixing a spurious ~0.8% "Missing" on perfect captures); dropout gaps and tight doublets fall outside the window, so the estimate keeps the median's robustness. **Δt is never persisted** — every stream re-detects its own (registers run at different rates); a manually typed+applied Δt holds only for the capture on screen.
- **Gap threshold** — multiplier of Δt above which a gap counts as a dropout (default 1.5×).
- **Too-close threshold** — a **multiplier of Δt** (default 0.9×) below which a *strictly positive* spacing is flagged as suspiciously tight. Source of truth is `closeFactor` (persisted); the ns value the analyzer/renderers consume (`closeNs = closeFactor × intervalNs`) is recomputed **inside `analyze`** (so a type override is honored) and at every load/Apply/auto-detect. A **0-gap is a duplicate, not "too close"** (the tight test is `0 < g < closeNs`), so the two never double-count. **Pontual streams have no cadence ⇒ `closeNs = 0`** in `analyze`, which suppresses every too-close marker/readout for them.
- **The six anomaly types** — surfaced consistently in both the single-stream view and the device-card scan: **missing** messages (within dropouts), **drop events** (count of gaps > tol·Δt), **too-close** (`0<g<closeNs`), **out-of-order** (`g<0` — a timestamp earlier than its predecessor), **duplicate** (`g===0` — same timestamp as predecessor), and **unparseable lines** (CSV rows whose first field isn't a timestamp). Missing/drop/too-close need a cadence (skipped for pontual); out-of-order/duplicate/unparseable apply to every stream.
- **Duplicate & out-of-order on the graph** — both are threaded through the **LOD pyramid** and `columns()` as per-column counts (`dup`/`ooo`), so they render at any zoom even when counts run to millions. On the trace (`drawDupOoo`) they're **cyan dots above / magenta dots below** the baseline; each gets a **count label** (dup above, ooo below) shown only when the marked columns are far enough apart that no two labels collide (`ctx.measureText`-gated). Each **out-of-order dot also gets an arrow** to where the sample would sit to be in order (its predecessor's timestamp; `drawOooArrows` over `oooEvents`, binary-searched via `visibleOooRange`, drawn only when ≤80 are on screen). The heatmap marks dup with a top-edge cap and ooo with a bottom-edge cap. Hover (trace + heatmap) reports the cursor column's dup/ooo with error-specific text. Legend entries (round swatches) on both legends.
- **Time modes** — absolute (epoch/wall-clock UTC) vs. relative (elapsed seconds), auto-detected: a first sample beyond ~1973 in epoch terms is treated as absolute. **Harp-epoch CSVs**: a numeric-seconds first value past ~3.0e9 s (≈1999 in 1904 terms ≈2065 as Unix) is treated as seconds-since-1904 and shifted to the Unix epoch (`HARP_EPOCH_OFFSET_NS`), mirroring the binary parser's rule — camera CSVs read correct 2026 wall-clock. Offsets stay relative to the raw first value.
- **Nanosecond precision** — timestamps are parsed to exact ns using BigInt; offsets from the first sample are stored as Numbers.
- **Trace layers** — three independent toggles drive the trace canvas: `showRaw` (per-channel value plots), `showAnom` (dropout/too-close spikes + heatmap + tables), and `showTs` (a vertical line per sample). They compose; `showTs` lines are drawn under the anomaly/raw layers and collapse to one line per pixel column via the same `columns()` aggregation the spikes use.
- **Measurement arrows** — on hover, double-headed arrows on the overlay canvas show the Δt between the items bracketing the cursor: between events (when `showAnom`, accent blue) and/or between samples (when `showTs`, steel blue).
- **Per-missing-message rendering** — a dropout is drawn as one line per missing message at its expected arrival time (`gap.start + k·Δt`), colored/height-scaled by the parent gap's severity; below ~2px expected spacing the burst collapses to one spike per pixel column. **Pontual streams draw no missing spikes at any zoom** — the collapsed (`col.maxGap > threshold`) path is gated on `!pontual`, matching the zoomed-in `gapEvents` path (empty for pontual); the hover (both the `showAnom` gap line and the `showTs` est-missing line) and the heatmap readout are gated the same way.
- **Missing-count model** — `analyze` counts missing as the **per-event sum** `missSum` (= Σ `max(1, round(g/Δt)−1)` over dropouts) for **both constant and burst** streams, with `expected = n + missSum`, `dropped = missSum`. This guarantees **missing ≥ dropout count** and matches the device-card scan (`makeStatsAcc`). (The old constant-only `expected = round(span/Δt)+1`, `dropped = expected − n` global estimate was dropped: rounding and duplicate/extra samples could make it fall *below* the dropout count.)
- **Settings persistence** — `tolerance`, `closeFactor`, and the three view toggles are saved to `localStorage` (`gapscope.settings`). Legacy keys (`closeNs` in µs, pinned `intervalNs`, `chunkMB`) are ignored on load. The **auto** button re-detects Δt on demand. **No Apply button**: the Δt / gap-threshold / too-close inputs apply immediately on change (`applySettings`); the controls header is shown only in the single-stream analysis view.
- **Per-experiment Visualizer config** — the Visualizer selection (value lines + camera videos) and the **Normalize each line** toggle are persisted to `localStorage["gapscope.visualizers"]` as a map keyed by `experiment.name` (folder name). `saveVizConfig` writes it on **Visualize** (`visualizeChart`) and whenever the norm toggle changes; `restoreChartSel` rebuilds `chartSel` from it when **＋ New Visualizer** is started (`enterChartEdit` pre-fills instead of clearing), validating that each device/stream/channel/video still exists (stale entries skipped — streams are keyed by their index within the device, stable for unchanged folder contents). One config per folder; re-visualizing overwrites it (no explicit clear yet).
- **Visualizer (combined value chart + optional video)** — a cross-device overlay of register/CSV *values* (no timing/missing analysis), optionally with one **or more** synchronized camera videos. In an experiment, **＋ New Visualizer** (`enterChartEdit`) enters a selection mode; clicking a register row / array position / CSV column on any card toggles it into `chartSel`; clicking any device's **🎞 video** row toggles a `kind:"video"` entry (**multiple allowed** — `videoKey`/`isVideoSelected`/`toggleVideoSel`/`selectedVideoDevs` returns them in card order). Selected rows get a **strong** highlight (`tr.csel`). **Visualize** (`visualizeChart`) loads each selected value stream (`loadChartStream`/`loadChartCsv`, decimated to `CHART_MAX_PTS`=300k, anchored to the experiment start), shows `#chartview`, and — if any videos were selected — attaches them via `vidAttach(videoDevs, "chart")`. A **video-only** Visualizer is allowed (empty plot + playhead; the video opens large). Lines are plotted vs the shared experiment time axis; X navigation mirrors the trace (wheel-zoom, **left-drag zoom-select**, right-drag pan, dbl-click reset; a small left-click with video active **seeks the video**). Y: per-line normalized by default — each visible line gets its own colour-matched gutter axis with high value at the top, low at the bottom; a **Normalize each line** toggle switches to one shared raw axis; >6 lines fall back to a single 0–100% axis. Legend entries are **clickable to show/hide** a line (same for the trace's raw-channel legend). Hovering shows a popup with the cursor time + each visible line's value. **Burst** streams break the line across their inter-burst silences instead of connecting — the silence-boundary indices are **precomputed once per line** in `visualizeChart` (a `breaks` array of sample indices where the loaded-offset gap ≥ the floor) and the draw loop lifts the pen whenever a boundary falls inside the decimation `step` interval, so the breaks are **identical at every zoom level**. (The old test compared the gap between consecutive *drawn* samples to `silenceNs`; because `step` grows when zoomed out it inflated that gap past the floor, so bursts that looked separated zoomed-out re-connected zoomed-in.) The break floor is **re-derived per line from the samples actually loaded** (`streamRegularity`/`detectInterval` over their real gaps), falling back to the background scan's `stats.silenceNs` when there's no clear bimodal silence band — so it isn't limited to the scan's first-65k-gap estimate. The **graph is vertically resizable** (`chartH`, default 360, drag handle `#chart-vresize` under the chart → `drawChart`+playhead redraw). The Visualizer's controls (＋ New Visualizer / Visualize / Cancel) live as **icon buttons in the top header nav** (`updateHeaderNav` shows them on the device-list page; `#chartcount` is a count badge on the Visualize button). `chartData`/`cvBase`/`cvStart`/`cvEnd`/`cvFull`/`cvSelRect`/`chartGeom`/`chartH` hold chart view state.
- **Camera video playback (WebCodecs, multi-camera)** — a device folder may hold H.264-in-AVI video (`.avi`, collected into `dev.videos`), paired by base name with a per-frame timestamp CSV. GapScope plays it **in-browser, no conversion** via the browser's `VideoDecoder` (`vidHasCodec` gate — Chrome/Edge only; configured `prefer-hardware` with a software fallback). **Each camera is a `makeVidPlayer(dev, idx)` instance** holding its own decoder + clip set + target canvas + colour; `vidPlayers[]` holds 1 (trace) or N (Visualizer). A **single shared clock** (`vidPosOff` ns-from-origin, `vidPlaying`/`vidSpeed`/`vidSeekHold`) and one transport drive every player; `vidTick` is the one rAF loop — it advances the clock, then calls each `player.drive()` and only releases the seek-hold once **all** players are `settled` (on their target frame, or in a gap). Demux helpers are free functions (`vidScanTopLevel` RIFF walk → `vidParseHdrl` fps/dims → `vidReadIndex` `idx1` byte-offset+keyframe per frame, auto-detected base; `vidDemuxHeader`/`vidLoadFrameOffsets`/`vidClipBounds`); per-clip routing/decoding is per-player (`clipIndexForOff`/`locate`/`seekDecode`/`driveDecode`/`pump`/`drawFrame`/`drawGap`/`activateClip`). Per-frame offsets come from the paired CSV anchored to `vidOriginNs()` (= experiment timeline start, the **same origin** the chart lines use), so the playhead is frame-accurate across dropped frames and across hourly clips (per-player `clips` sorted by start, swapped on seek). Decoder emits in **presentation order**; feeds from the nearest keyframe `vidKeyAtOrBefore` (GOP≈250). **Seeking** (`vidSeekOff`) holds the playhead and calls `p.markSeek()` on each player (one reseek per seek); a **paused** seek `flush()`es to drain held B-frames, and a flushed decoder is never re-fed per-tick (reseeked only on the next real seek/resume — reseeking on flush looped the catch-up forever). **Coverage gaps** — `p.locate(off)` flags lead-in / tail / between-clip gaps (>`VID_GAP_TOL`=1s, so the ~25ms clip seams don't count); a player in a gap draws a "no footage" placeholder (`drawGap`) and counts as settled. A clip that fails to demux is marked `loadFailed` and treated as a gap so one bad file can't freeze the shared clock. `vidMode` ("trace" | "chart") picks the panel set — trace reuses the static `#video-block`/`#vid-canvas`; chart builds a **responsive grid** (`#chart-vidgrid` → `.vidpanel` per camera, colour-keyed caption + canvas, `vidBuildPanels`). Trace and Visualizer never coexist. A standalone single-camera prototype lives in [video_test.html](video_test.html).
- **Video transport controls & high-speed scrub** — speed options span **0.1×–128×** (both the trace `#vid-speed` and Visualizer `#chart-vid-speed` selects). The **←/→ keys skip ±5 s** (`vidSkip` → relative to the live clock position → `vidSeekOff`; `VID_SKIP_NS`=5e9; ignored while typing in an input/select). **Keyframe-only scrub** keeps fast playback tracking the playhead: per player, when the displayed frame falls more than `VID_SCRUB_LAG` (60) frames behind the target for `VID_SCRUB_TICKS` (6) consecutive ticks during *free* playback (dynamic lag detection in `drive`), it switches to `scrubActive` (`enterScrub` makes a clean decoder fed **only keyframes**; `scrubStep` feeds the nearest keyframe ≤ target, one in flight via `scrubBusy`, `flush()`ing to force it out; the `output` callback draws scrub frames straight to the canvas). Keyframes are self-contained, so this is correct **despite B-frames** (decode order ≠ presentation order) and needs **no per-frame decoder recreation**. Any pause/seek/decoder-error/gap or speed change calls `exitScrub` (→ `markSeek`) to rebuild precise contiguous decode. **Two earlier approaches were tried and reverted**: (a) recreating the decoder per skip churned the hardware decoder and **crashed the GPU process**; (b) tagging frames by their fed (decode-order) timestamp broke playback at *every* speed because B-frame reordering means decode order ≠ presentation order — the working normal path counts decoder outputs (presentation order) instead.
- **Visualizer video UI — focus, resize, ROI loupe** (chart mode only): the seek **playhead is orange**. **Click a camera** in a multi-camera grid to **focus** it (enlarge, hide the others — `vidFocusCam`/`vidApplyFocus`, focus-only; clicking the enlarged one does nothing); return via the **⤢ All cameras** button (`#chart-vid-all`) or **Esc**. A **solo** video (single-camera Visualizer, or the focused camera — `vidSoloPlayer`/`vidUpdateSolo`) is **freely resizable** keeping aspect (width-driven via the `--solo-w` CSS var + a corner grip `.vid-rsz`/`vidRszStart`; height auto). **ROI loupe** (`vidOvlDown`/`vidDrawOverlay`/`vidUpdateLoupe`/`vidFitSolo`): drag a **square** on the solo video (a transparent overlay canvas `.vid-ovl` captures it; box is **movable + resizable** by corner) to see that region **zoomed in a panel to the right** (`.vidloupe`), sized to the video's height, refreshed live from the native-res frame each `drawFrame`/`drawGap`. The grid switches from `display:grid` to **flex** while the loupe is open (set inline in `vidShowLoupe`, since `vidShowPanel` writes an inline `display:grid` that a stylesheet rule can't beat), and `vidFitSolo` caps the video width so the full-height loupe fits beside it (`videoW·(1+aspect) ≤ container`). ROI is stored in **frame pixels** (survives resize). Clear with the loupe **✕** or **Esc** (Esc order: clear ROI → unfocus → back to devices). All ROI/focus state resets in `vidDetach` and when the solo target changes.
- **Video coverage overlay** — `vidDrawCoverage` (called at the end of `drawTrace` and `drawChart`) marks where footage exists: **one coverage track per camera** stacked at the top edge (each in its player colour, solid where clips have frames), plus a full-height **dim of the times no camera covers** (union of all players' spans). Pixel-space span merging (`vidMergePx`, <1px gap) collapses sub-pixel clip seams while keeping real lead-in/tail gaps visible. Single-camera keeps the "video" label; multi relies on the colour↔panel mapping. Redrawn when clip bounds resolve (`vidAttach` calls `drawChart`/`scheduleDraw`).
- **Time-axis labels** — elapsed time renders as a **composite** `d/h/m/s/ms/µs` string (`fmtElapsedNs`, e.g. `1h 20m 27s 088ms`), smallest unit following the tick step (`niceTimeStep` over 1 µs … 10 days, `resIndexForStep`). When the experiment timeline starts in the 2020s or later (`expAbsolute`, a real wall clock), the combined chart's X axis instead shows **absolute UTC** — ticks land on round clock times (`fmtClockTick`) and the hover/summary show full stamps (`fmtClockFull`). The single-stream trace shows the composite elapsed label alongside the wall clock.
- **Zoom depth** — tightest window is `intervalNs * 5` (~5 samples), uniform regardless of capture size.
- **Chunked streaming load** — files are read as **fixed 8 MB** (`CHUNK_BYTES`) byte slices via `file.slice` + `FileReader.readAsArrayBuffer`, decoded with a streaming `TextDecoder` carrying the partial trailing line, fed line-by-line into `makeLineParser().feed()`. Max file size is bounded by sample-count memory, not raw bytes. The loaded meta line reports a `loaded in …` time. (The old Read-chunk control was removed.)
- **Harp binary format** — `.bin` files are a flat stream of fixed-stride messages `[Type, Length, Address, Port, PayloadType, (Timestamp), Payload…, Checksum]`; `stride = Length + 2`, `PayloadType` carries the dtype code with bit `0x10` set when a timestamp is present, timestamp is `uint32 s + uint16 32µs-ticks` since **1904-01-01**. A single `.bin` holds one register; stride/dtype/channels lock in from the first message. `makeHarpParser` fills the **same globals** as `makeLineParser`, so analyze/pyramid/render are format-agnostic. Clock-synced captures (first second past ~1999) get the 1904→1970 shift. Format chosen by `detectBinary` (extension, else printable-byte sniff).
- **Register snapshot dumps** — a `.bin` whose messages span *different* addresses (e.g. `Behavior0_<ts>.bin`, no register number) is the per-register device state written at connect: one message per register, variable stride. Detected in `readHarpHeader`'s pre-scan (second message address differs); excluded from register grouping; parsed whole by `parseHarpSnapshot` (variable-stride walk) for the **register inventory** and **WhoAmI** (register 0) — used to identify the device.
- **Known-device schema library** — dropped `device.yml` schemas (any `.yml` that parses to `device`/`whoAmI`/`registers`) join `deviceSchemas` keyed by device name and are **persisted as raw yml text** in `localStorage["gapscope.devices"]` — drop the `devices/` folder once and it's remembered across sessions. File-name fallback (when `device:` is absent) honors the `<device>.device.yml` convention. The drop zone shows "N known device schemas … forget". `matchSchema(name, whoAmI)`: exact **WhoAmI** match first (from a snapshot dump), else the **longest case-insensitive device-name prefix** of the folder/file name (`Behavior` → `Behavior0`). `harpSchema` is the schema applied to the *current* load ({device, whoAmI, registers:{addr→{name,members}}}); a schema dropped on its own also becomes active (old sticky behavior). Ymls that don't parse as device schemas (e.g. a rig configuration) are treated as data/config, never as schemas; `.md`/`.json` riding along a flat drop are inert.
- **Folder classification** — every dropped/picked file carries a relative path (`webkitRelativePath` from the picker; drag traversal stamps `entry.fullPath` as `_relPath`; bare files have none). `splitByFolders` strips the dropped folder itself (shared first segment), then: files in sub-folders → **experiment** (devices keyed by first segment; root-level files stashed as `configFiles` — `Metadata.json`, `RigConfiguration_*.yaml`, currently unused but kept for the rig-mapping feature); all files at one level → **flat** (the original single-device behavior: bins grouped by register, or one CSV).
- **Experiment mode** — `experiment = {name, configFiles, devices[], timeline}`; a device is `{name, binFiles, csvs, other, bytes, byAddr, addresses, snapshot, schema, streams, _card}`. On drop, every device's bins are header-scanned (first KB: register grouping + snapshot detection) and snapshots parsed; then `computeTimeline` measures the shared experiment span before the cards render. The **device card grid** shows: name, schema badge (matched name or "unknown device"), snapshot register count + WhoAmI, sizes, and a per-stream table (**type chip** / msgs / Δt / **⚠ potential-problems**) that fills in as the background scan completes. The last column replaces the old "missing" cell with the **total count of all six anomaly types** (`streamProblems(st)`); hovering a non-zero count shows a **modern floating popup** (`#prob-ro`, reusing the `.readout` look — dark rounded card, fade-in, cursor-following, fixed-positioned over the grid) with one colored line per problem (palette matches the trace: missing=red, too-close=violet, duplicate=cyan, out-of-order=magenta, unparseable=amber), the unparseable line carrying its file + first-error line. Δt shows `—` for pontual streams (no cadence). The breakdown rides on the cell as a `data-prob` attribute (`encodeURIComponent(JSON)`); delegated `mouseover/mousemove/mouseleave` handlers on `#devgrid` drive the popup. Clicking anywhere on a card opens the largest-by-bytes stream; clicking a specific **register row** (`tr[data-i]`, hover-highlighted) opens that stream directly — `openDevice(dev, idx)`. A **"← devices"** back button returns; "Load another capture" or a flat load closes the experiment; schema imports while it's open re-match unknown devices live.
- **Streams** — the loadable units of a selection: one per Harp register (its files concatenated in timestamp order through a single parser → one continuous timeline) plus one per **CSV base name** — hourly chunk files `<base>_<timestamp>.csv` are grouped by `csvBaseName`/`groupCsvs` and concatenated in name (time) order through one `makeLineParser` via `streamParseTextFiles`, so a split CSV reads as one stream. A CSV stream carries `nEl`/`cols` from a first-KB header peek (`readCsvHeader`); a Harp register's `nEl` comes from `readHarpHeader`. `streamChannelNames(stream, schema)` resolves the per-value-line names (device.yml members / CSV header columns / fallback). On a device card an **array register or multi-column CSV expands** (caret, collapsed by default, state on `dev._expanded`) into one sub-row per value. The loaded-bar selector (`#regsel`) is a **custom dropdown** (native `<select>` can't render the colored type chip): a button showing the current stream's chip+label and a popup listing every stream with its type chip on the left (`refreshStreamSelector`, rebuilt at the end of every `render` so the chip tracks the resolved type/override). Shows when there are ≥2 streams; default is the largest by bytes. Flat multi-bin drops use the same machinery (`buildStreams`/`loadStream`).
- **Background summary scan** — after the device list renders, a sequential queue walks every stream with **timestamp-only** parsers (`makeHarpStatsWalker`/`makeCsvStatsWalker` → `makeStatsAcc(spanNs)`): first 65,536 gaps buffered → `detectInterval` + `streamRegularity` → every gap judged against `tolerance × Δt`, tracking the full and silence-attributable subsets; `done()` classifies via coverage (`count·Δt / experiment span`) and yields {count, dtNs, drops, missing, type, **outOfOrder, duplicates, tight, malformed, malformedFiles**} (burst figures exclude in-data silences; `tight` is 0 for pontual; `malformedFiles` = `[{name, count, firstLine}]` attributing unparseable CSV rows to their file + first-error line, via `acc.file(name)` boundary marks) with O(1) memory, never touching the loaded capture's globals. **Paused** while a device is open (in-flight chunk parks its continuation on `scanState.cont`), resumed on back, cancelled on close (epoch counter orphans in-flight reads). `#scanline` shows progress. Card numbers are estimates (Δt from the first 65k gaps, tolerance as of scan time) — opening the device computes exact figures.
- **Big-capture render perf** — per-frame cost proportional to what's on screen: `columns`/`channelColumns` bind to visible samples via `lowerBound`; the full-capture heatmap columns are cached (`heatCache`/`heatEpoch`); dropout loops use `visibleGapRange` (binary search over `gapEvents`).
- **LOD aggregate pyramid** — `buildPyramid()` (once per load) precomputes multi-resolution clusters (finest bins `LOD_BASE`=64 raw samples; coarser levels merge pairs). `pickLevel` chooses the coarsest level with `size ≤ samples/pixel`; `columns`/`channelColumns` aggregate from it via `firstBin`, making zoomed-out draws O(cols). Heatmap stays pixel-exact (`forceRaw`). Built when `N ≥ LOD_BASE*4`.
- **Trace navigation** — scroll = wheel-zoom at cursor; **left-drag = zoom-select**; **right-drag = pan**; double-click = reset. The heatmap minimap has its own click/drag navigation.
- **Known browser limit** — folder *drag-drop* traversal is blocked by Chromium on `file://` pages; a folder drag from disk errors with a message pointing to the **"pick a folder"** button (not restricted) or to serving over `http://localhost`.

## Ideas / possible future work

- **RigConfiguration mapping (NEXT — agreed with the user)** — parse the experiment's `RigConfiguration_*.yaml` / `Metadata.json` (already stashed in `experiment.configFiles`) to map *role* folder names (LoomEast, CameraController, Feeder0…) to device types, so schema matching covers devices that neither prefix-match nor carry a snapshot WhoAmI.
- **Cross-register merge** — the combined value chart now overlays *values* from different registers/devices on the shared timeline (values only). A merge for *gap analysis* across registers (a union timeline with mixed rates) is still open.
- **Combined chart polish** — CSV columns are selectable; possible follow-ups: align absolute X ticks more precisely at deep µs zoom, a "select all positions" affordance, persisting a chart selection.
- **Schema/data mismatch note** — surface a small UI warning when a register's declared `device.yml` member count differs from the channel count found in the `.bin`.
- **`device.yml` `$ref`/`!include`** — follow external schema references (e.g. `common.yml`) that the minimal YAML reader skips.
- **Re-scan on threshold change** — device-card stats keep the tolerance from scan time; changing the Gap threshold could invalidate/re-run the summary scan.
- **Multi-camera performance** — DONE: multiple cameras play side by side (one `makeVidPlayer` per camera, shared clock). Decoders now request `prefer-hardware`. Open: with many large (2448×2048) cameras at once, decode may still stutter — possible follow-ups are capping concurrent cameras, only feeding decoders near the playhead, or a quality/throttle control. Per-camera coverage tracks already make it cheap to see which cameras matter at a given time.
- **OpenDML / >1 GB AVI** — the demuxer uses the classic `idx1` index; files >1 GB (or with only the OpenDML `indx` index) aren't handled yet.
